# Ресёрч: чего не хватает редактору

Дата: 2026-06-13. Источники:

- 82 официальных шаблона игры (`StreamingAssets/map_templates`, полная инвентаризация полей — `scripts/analyzeFieldCoverage.mjs`);
- [Olden-Era---Template-Generator](https://github.com/KhanDevelopsGames/Olden-Era---Template-Generator/) (Khan) — WPF-генератор шаблонов;
- [AuroraRMG](https://github.com/sany86russ/AuroraRMG) (sany86russ) — развитие генератора Khan: 43 пресета, баны, анализ баланса.

Числа в скобках — частота поля в официальных шаблонах (сколько объектов его используют).

## 1. Потери данных при импорте/экспорте — чинить в первую очередь

Поля, которые игра использует, а наш экспорт **молча подменяет**:

- [x] **`gameMode`** — экспорт хардкодит `"Classic"`, в игре есть шаблоны `"SingleHero"` (82 шаблона имеют поле). *(сделано)*
- [x] **`gameRules.heroHireBan`** — хардкод `false`, в части шаблонов `true` (запрет найма героев). *(сделано)*
- [x] **`gameRules.encounterHoles`** — хардкод `false`, в части шаблонов `true` («дыры» в охране — проход без боя). *(сделано)*
- [x] **`variants[].border`** — вообще не импортируется, на экспорт всегда уходит дефолт. Теряются: **вода по краям карты** (`waterWidth`, `waterType`), скругление углов (`cornerRadius`), ширина и шум кромки препятствий (61 из 92 вариантов задают border явно). *(сделано: passthrough по-вариантно + UI вода/скругление)*

## 2. Поля формата: сохраняются (passthrough), но не редактируются

### Правила игры

- [x] **Стартовые бонусы** (`gameRules.bonuses`, 38 шаблонов) — ресурсы/заклинания/статы герою на старте. В Aurora есть полноценный пикер. *(сделано: секция «Бонусы старта», 8 типов из каталога map_bonuses; детали — research-bonuses.md)*
- [x] **Баны** (`globalBans`, 30+ шаблонов): предметы (`items`), заклинания (`magics`), герои (`heroes`). У нас уже загружен Core.zip — можно показывать реальные имена, как делает Aurora. *(сделано: секция «Баны» с каталогом 108 героев / 131 заклинания / артефактов)*
- [x] **Множители опыта** (`factionLawsExpModifier`, `astrologyExpModifier`, ~40 шаблонов) — два слайдера. *(сделано: поля в общих настройках; раньше значения молча терялись при импорте)*
- [x] **`valueOverrides`** (26 шаблонов) — переопределение ценности/охраны конкретных объектов. *(сделано: секция «Охрана объектов» в левой панели)*

### Зоны

- [ ] **`guardMultiplier`** (996 зон) — общий множитель силы охраны зоны. У Khan/Aurora это слайдер «Neutral army strength».
- [ ] **`diplomacyModifier`** (593) — «сговорчивость» нейтралов (присоединение/бегство). В Aurora — слайдер −1…+1.
- [ ] **`guardReactionDistribution`** (1182) — агрессивность охраны. В Aurora — пресеты «Пассивные/Обычные/Агрессивные».
- [ ] **`guardCutoffValue`**, **`guardRandomization`**, **`guardWeeklyIncrement`** на уровне зоны (1182) — у нас есть только на связях.
- [x] **Параметры рельефа** (`zoneLayouts`: `obstaclesFill`, `lakesFill`, `elevation*`, 278 лэйаутов) — плотность препятствий, озёра, перепады высот. В Aurora — слайдеры «Terrain roughness» / «Lake amount». У нас лэйауты сохранялись только passthrough. *(сделано: секция «Профили рельефа» в левой панели — разделяемые профили как в формате игры, селект профиля у зоны; распределения наград passthrough)*
- [x] **Per-area ценности** (`guardedContentValuePerArea` и пары к ним) — прирост ценности от площади зоны. *(сделано: секция «Прирост ценности от площади» у зоны; раньше значения официальных шаблонов молча обнулялись при экспорте)*
- [x] **`contentBiome` / `metaObjectsBiome`** — биом контента и мета-объектов отдельно от биома земли (мы редактируем только `zoneBiome`). *(сделано: два селекта «Как у земли»+режимы в секции биома; экзотические правила остаются passthrough)*
- [x] **Тип дорог** (`zone.roads[].type`: `Stone`/`Dirt`, 2616 сегментов) — у нас дорога есть/нет, тип не выбирается. *(сделано: селект «Покрытие дороги» на связи; нетипизированные сегменты сохраняются как есть)*
- [ ] **`crossroadsPosition`**, **`randomHire*`** — нишевые, низкий приоритет (passthrough). **`encounterHolesSettings`** — *(сделано: чекбокс + два поля в «Охране и нейтралах» зоны)*.

### Главные объекты зоны

- [x] **Типы помимо Spawn/City**: **`AbandonedOutpost`** (форпост — «remote footholds» в Khan/Aurora) и **`GladiatorArena`** (объект арены). Сейчас наш редактор их не создаёт. *(сделано: кнопки в инспекторе зоны, импорт перестал их выбрасывать)*
- [x] **`owner`** (41) — город, изначально принадлежащий игроку (в т.ч. нейтральные зоны под владением). *(сделано: селект «Владелец» у города)*
- [x] **`placement` / `placementArgs`** (1020/456) — правило размещения объекта в зоне (Uniform/Center/Connection/NearZone…). *(сделано: секция «Размещение в зоне» на карточке главного объекта; пресеты аргументов из официальной статистики, выбор связи/зоны, валидация ссылок; неизвестные значения остаются passthrough; enum MainObjectPlacement и поле isCenter подтверждены по il2cpp-метаданным игры)*
- [x] **`buildingsConstructionSid`** (839) — стартовая отстройка города (default/poor/rich…). *(сделано: селект «Стартовая отстройка» у замка/спавна/форпоста)*
- [x] **Охрана города**: `guardValue`, `guardChance`, `guardWeeklyIncrement`, `removeGuardIfHasOwner` у City/Outpost. *(сделано: секция «Охрана объекта» на карточке; экспорт больше не навязывает 40000/0.1 объектам без охраны)*

### Связи

- [x] **`guardZone`** (275) — в какой зоне стоит охрана прохода. *(сделано: «Тонкая настройка прохода» в инспекторе связи)*
- [x] **`gatePlacement`** (206) — размещение «ворот». *(сделано: там же)*
- [x] **`guardMatchGroup`** (107) — связи с одинаковой (синхронной) охраной. *(сделано: там же, с подсказкой существующих групп)*
- [ ] **`portalPlacementRules`** (34) — конструктор правил размещения порталов (решено оставить passthrough до реального спроса).

### Контент

- [x] **Редактор `contentPools`**: распределение ценности (`valueDistribution.priceBounds/weights`), группы с весами, баны пула. *(сделано: общие именованные пулы — секция в левой панели, инспектор с распределением по ценам, группами из каталога списков и банами; три слота ссылок в инспекторе зоны с «Авто» по типу; группы с `content`-уточнениями — verbatim passthrough; авто-пулы visual_pool_* остаются фолбэком для зон без явных ссылок)*
- [x] **Редактор `contentCountLimits`**: лимиты количества объектов (`maxCount` по sid/variant/спискам) — 12 тысяч записей в официальных шаблонах, активно используемая ручка баланса. *(сделано: общие именованные списки по образцу профилей рельефа — секция в левой панели, инспектор с каталогом, ссылки в инспекторе зоны с «Авто» по типу; редактируемые формы {sid[,variant],maxCount} и {includeLists,maxCount}, экзотика с `content` — verbatim passthrough; playerMin/playerMax; встроенные стабы content_limits_spawn/side)*
- [ ] **Тонкости mandatoryContent**: `designatedEncounter`, `owner`, вложенные списки с весами, полный конструктор правил размещения (`rules`). Базовые правила у нас уже есть через пресеты дистанций.

## 3. Фичи инструментов-конкурентов (не поля формата)

### Есть у обоих (Khan и Aurora)

- [x] **Генерация топологий**: Random (Делоне), Ring, Chain, Hub, у Aurora ещё Balanced (кольца по качеству) и Lanes (коридоры). У нас граф строится только вручную — просится «мастер генерации скелета», результат которого дальше правится на холсте. *(сделано: кнопка «Мастер» в топбаре — кольцо/цепь/звезда/случайная, изоляция игроков, пресеты по ролям, ценности сразу с масштабом от площади)*
- [ ] **PNG-превью карты**, у Khan — **сохранение превью рядом с шаблоном для показа в игре**. Стоит проверить механизм и поддержать экспорт изображения холста.
- [x] **Автопоиск папки игры** (через реестр Steam) и сохранение сразу в `map_templates`. Для веб-приложения частично достижимо через File System Access API (выбрать папку один раз — потом сохранять в один клик). *(сделано: кнопка «В игру» — папка выбирается один раз, хэндл персистится; подтверждение перезаписи; импорт шаблона добавляет « (copy)» к имени)*

### Только Aurora

- [x] **43 готовых пресета шаблонов** (1v1, FFA 3–8, PvE 1×2…1×7, спецрежимы) — мгновенный старт. У нас есть пресеты зон, но нет пресетов целых шаблонов. *(сделано: кнопка «Шаблоны» — 10 курируемых архетипов вместо 47 комбинаций; вариации через мастер топологий)*
- [x] **Анализ баланса** после генерации: оценка 0–100 + сводка «что внутри» (зоны/связи/сокровища). У нас есть валидация корректности, но не баланса. *(сделано: секция в правой панели — балл, симметрия богатства/расстояний/охраны выходов по игрокам, структурные находки; живой пересчёт)*
- [ ] **Simple/Advanced режимы** — упрощённый интерфейс для новичков.
- [ ] **Большие карты до 512×512** (экспериментально; игроки реально играют 300–400). У нас селектор размеров + произвольный ввод — стоит проверить верхние границы.

### Только Khan

- [x] **Isolate player zones** — гарантия «игроки соединены только через нейтралов» (у Aurora тоже есть как опция «Connect only via neutral zones»). *(сделано: чекбокс в мастере топологий)*
- [ ] **Match player castle factions** — замок фракции игрока в стартовой зоне. У нас фракция города выбирается, но нет режима «по фракции игрока в лобби» (в формате это `faction: {type: Match, args:["0"]}` — 458 объектов используют Match-правила; у нас поддержан ли Match — проверить полноту режимов фракции).

## 4. Что у нас уже сильнее обоих

Для контекста: визуальный граф с drag-and-drop и дугами, **импорт** существующих шаблонов (оба конкурента только генерируют с нуля), точный round-trip с passthrough незнакомых полей, варианты генерации, инспектор связей по парам, пользовательские списки объектов и пресеты зон, живая валидация и JSON-превью, RU/EN, тёмная/светлая темы, автосохранение.

## 5. Рекомендуемый порядок работ

1. **Фиксы потерь** (раздел 1) — дёшево, повышает фидельность: gameMode, heroHireBan, encounterHoles как настройки; border — минимум passthrough, максимум UI «вода/границы».
2. **Баны и бонусы** (UI поверх Core.zip — каталог уже загружен).
3. **Ручки баланса зоны**: агрессия охраны, сговорчивость нейтралов, множитель охраны, рельеф/озёра.
4. **Главные объекты**: типы Outpost/Arena, владелец, отстройка города.
5. **PNG-превью** (+ выяснить in-game preview).
6. **Мастер топологий** (генерация скелета с последующей ручной правкой — наше преимущество против «чистых генераторов»).
7. **Пресеты целых шаблонов**.
8. **Анализ баланса** и редакторы пулов/лимитов — длинный хвост.

## 6. Сверка с декомпилированными исходниками игры (2026-06, через доки derpcatmusic/HoMM_OE_RMG_Editor)

Их `docs/rmg/*` выверены против декомпилированного `RandomMapTemplate.cs` / `MapGenerator`. Итоги сверки нашего кода:

**Исправлено по итогам:**
- [x] **`factions` (массив) — мёртвое поле**: движок читает кандидатов фракции только из `faction.args` (FromList). Наш экспорт «конкретной фракции» эмитил `faction: {FromList, args: []}` + `factions: [id]` → в игре давало случайную фракцию. Теперь эмитим `args: [id]`.
- [x] **Порядок весов реакции охраны**: enum игры — `Aggressive, Negative, Common, Friendly, Peaceful, Docile` (0-й = всегда бой, 5-й = присоединяются/пропускают). Наши пресеты и подписи предполагали обратный порядок — пресеты пересобраны, у слотов появились тултипы с именами реакций. Официальный дефолт [60,20,10,10,2,0] = преимущественно агрессивные.
- [x] **`guardRandomization` на связях — мёртвое поле** (есть у зон и главных объектов, в модели Connection отсутствует; 166 официальных использований игнорируются игрой). Убрано из UI связи, импорт-passthrough сохранён. Аналогично мёртвые на связях: `road` (917) — реальные дороги задаются сегментами `zone.roads` (мы их и так пишем), `simTurnSquad` (430).
- [x] **Валидатор**: `weights.length == priceBounds.length + 1` у пулов (блокирующая проверка игры); ровно 6 весов реакции.
- [x] **Подсказки**: perArea-формула (`итог = значение × проходимая площадь × 0.1`); из наборов лимитов/пулов зоны при генерации сэмплится ОДИН.

**Проверено — у нас уже корректно:** `cityHold`/`cityHoldDays`, гладиаторские поля (вкл. `gladiatorArenaRegistrationStartWork`, `championSelectRule`), строковая форма `contentCountLimits`, passthrough неизвестных полей (`uniqueMagicCostModifiers` и т.п.), `guardedEncounterResourceFractions` (countBounds+1 == fractions у встроенных профилей).

**Отложено (низкий приоритет):** UI для `gatePlacement: NearZone` + `gatePlacementArgs` (passthrough работает); предупреждение о дублях имён связей; `guardZone` с чужим именем зоны (наш UI таких не создаёт, движок молча игнорирует); дефолты ZoneLayoutConfig движка (obstaclesFill 0.4, lakesFill 0, minLakeArea 20, roadClusterArea 70) — мы всегда пишем поля явно, расхождение не влияет.

## 7. Финальная проверка перед публикацией (2026-06-12)

Сводка по всем источникам перед выкладкой на GitHub Pages:

1. **Файлы игры (84 официальных шаблона, StreamingAssets/map_templates)** — `npm run test:game`: 170/170 round-trip тестов проходят байт-в-байт; `scripts/analyzeFieldCoverage.mjs` не находит полей вне нашего инвентаря (root/zone/connection/mainObjects/mandatoryContent/contentCountLimits/contentPools/contentLists).
2. **Декомпилированная модель игры** (через доки derpcatmusic) — раздел 6: все блокирующие расхождения исправлены, выводы закреплены тестами (faction.args, 6 весов реакции, weights = priceBounds + 1, мёртвые поля Connection).
3. **Конкуренты** (Khan, AuroraRMG, OldenEraMaps, derpcatmusic) — фичегэпы из разделов 2–3 закрыты задачами #20–27 (placement, лимиты, пулы, Simple/Expert, баны, рецепты, анализ баланса).
4. **Кнопочный слой** — tests/exportCoverage.test.ts (модель → JSON) и tests/storeActions.test.ts (кнопка → модель) покрывают все model-shaping действия стора; tests/i18nConsistency.test.ts держит словари в паритете.
5. **E2E в превью** — применение рецепта через UI («Джебус-дуэль», «Грандиозная дуэль») даёт корректную структуру: связность, изоляция игроков, win_condition_1, пулы без висячих ссылок при отсутствующем Core.zip.

Итого: 302 теста зелёные (24 файла), tsc и eslint чистые. Известные ограничения остаются в списке «Отложено» раздела 6.
