# Ресёрч: переосмысление пресетов зон

Дата: 2026-06-11. Вопрос: пресеты зон сделаны «по-тупому» — фиксированные абсолютные значения, не масштабируются, не извлекаются из шаблонов. Как сделать правильно?

Источники: 82 официальных шаблона (1042 зоны), исходники AuroraRMG (полные), репозиторий Khan (исходников генератора в репозитории нет — только бинарь и data-файлы; анализ по Aurora, которая является развитием Khan).

---

## 1. Что сейчас у нас

`Preset` = `{ baseType, guardedValue, unguardedValue, resourcesValue, objects[], biomeMode }`:

- 6 захардкоженных дефолтов + пользовательские (`isCustom`, переживают импорт);
- значения абсолютные, без масштабирования и без perArea;
- пресет не знает про: профиль рельефа, охранные ручки зоны (guardMultiplier, агрессия...), perArea-ценности, главные объекты, пулы контента;
- «Сохранить зону как пресет» есть, но сохраняет только базовые поля.

## 2. Данные: официальные шаблоны

**Ценности почти не масштабируются от размера карты.** Средний guarded стартовой зоны: 146k на карте 96 и 146k на 128; 280k на 160 и 230k на 176. Корреляция слабая и немонотонная. Нормализация по формуле площади (см. §3) сжимает разброс p90/p10 лишь с 6.7× до 5.2× — **главная ось вариации не геометрия, а задуманная «богатость» шаблона** (бедный Jebus против жирной сокровищницы).

**Квантили guarded-ценности (все карты):**

| Роль | p10 | p25 | p50 | p75 | p90 |
|---|---|---|---|---|---|
| Стартовая | 75k | 125k | 240k | 350k | 500k |
| Открытая | 156k | 240k | 400k | 600k | 800k |

**Родной механизм масштабирования формата — поля `*PerArea`**: используются в 37% зон (guarded), добавляют ценность пропорционально фактической площади зоны. Типичные значения 200–3000/area.

**Размеры зон** дискретные: 0.5 (8%), 1 (77%), 1.5 (15%) — то есть сами авторы рулят площадью через size, а ценностями — руками + perArea.

## 3. Как у Aurora (наследник Khan)

Три слоя, сверху вниз:

1. **47 пресетов целых шаблонов** (`Presets.cs`) — «1v1 Классика», «PvE 1×7»… Задают размер карты, игроков, число зон, качество, топологию, плотности. Это НЕ пресеты зон.
2. **Тиры качества нейтральных зон** (`GetNeutralZoneProfile`) — Low/Medium/High, каждый тир это полный профиль зоны:

   | Параметр | Low | Medium | High |
   |---|---|---|---|
   | guardedContentValue | 120k | 240k | 480k |
   | guardedPerArea | 1000 | 2000 | 3000 |
   | unguarded / PerArea | 25k / 200 | 38k / 300 | 80k / 620 |
   | resources / PerArea | 30k / 240 | 55k / 420 | 90k / 580 |
   | пулы контента | t2 | t3 | t4+t5 |
   | guardMultiplier зоны | 1.1 | 1.4 | 1.8 |
   | профиль рельефа | sides | sides/treasures | treasures |
   | отстройка города | poor | rich/poor | rich |

   Тиры Aurora хорошо ложатся на наши квантили (p25/p50/p75-90 открытых зон: 240k/400k/600-800k).

   **Итоговая лестница встроенных пресетов редактора (ревизия 2026-06):** L «Проходная» 180k (бедно, под q25) → M «Середина» 400k (средне, q50–q75) → H «Сокровищница» 800k (оч. богато, q90); спавн 240k (официальная медиана стартовых), буфер N 120k. PerArea шагает вместе: 900/1800/3600 guarded, guardMultiplier 1.1/1.4/1.8. Имена в UI локализованы через `zoneTypeOption*`, список пресетов показывает подзаголовок с значением и квантильным бейджем.

   **Гарантированное наполнение (mandatoryContent) — частоты официальных шаблонов по ролям** (среднее на зону; 285 стартов / 633 проходных / 292 сокровищницы, порог отбора ≳0.5):
   - старт: pandora_box 1.26, mine_wood 1.11, mine_ore 1.11, watchtower 0.61, mine_gold 0.58, alchemy_lab 0.44, низкоуровневые жилища ~0.7;
   - проходная: pandora_box 0.80, mine_gold 0.61, chest 0.51, camp_fire 0.40, watchtower 0.38;
   - сокровищница: dragon_utopia 0.86, pandora_box 0.66, random_item_legendary 0.63, mine_gold 0.45, remote_foothold 0.25, unstable_ruins 0.22.

   Официальные зоны гарантируют в среднем 5–9 объектов; остальное добирают пулы по бюджету. Пресеты редактора приведены к этому скелету (5–8 гарантий на зону), артефактные списки и «декор» убраны из гарантий — их выдают пулы visual_pool_*.
3. **Масштабирование и плотности при генерации**:
   - `ContentScale = clamp(sqrt(zoneArea / 6400), 0.5, 2.5)`, где `zoneArea = mapSize² / totalZones`; абсолютные значения × scale, perArea × sqrt(scale). Эталон Aurora — 160×160 на 4 зоны (их генератор делает мало крупных зон; у официальных шаблонов типичная зона ~1300–2000 клеток, поэтому для нас эталон надо брать ~1600, иначе clamp срабатывает почти всегда).
   - Пользовательские проценты-слайдеры: ResourceDensity / StructureDensity / NeutralStackStrength (80–220%) — простые множители поверх.

## 4. Рекомендация

### 4.1. Авто-пресеты из каждого импорта — НЕ делать

82 шаблона × ~12.7 зон = тысяча почти уникальных комбинаций; автоматическое извлечение завалит список мусором. Уже работает: пользовательские пресеты и так переживают импорт .rmg.json (presets — app-уровень), а точечное «Сохранить зону как пресет» покрывает случай «понравилась зона из чужого шаблона». Опционально потом: кнопка «Извлечь пресеты из шаблона» с группировкой зон по сигнатуре и ручным выбором — но не автоматом при каждом импорте.

### 4.2. Новая модель пресета: роль × тир × мягкое масштабирование

**Пресет = роль + тир качества + полный профиль зоны:**

- **Роль**: стартовая / открытая (проходная) / сокровищница / центральная — определяет дефолтные главные объекты, профиль рельефа, агрессию охраны.
- **Тир**: бедная / средняя / богатая — таблица значений, откалиброванная по квантилям официальных данных (§2), а не от балды. Включая perArea-компоненты, guardMultiplier, отстройку.
- **Пресет тянет все поля зоны**, которые мы уже умеем редактировать: профиль рельефа (ссылка на профиль из секции «Профили рельефа»), охрана и нейтралы, perArea-ценности.

**Масштабирование — при применении пресета, не динамически:**

- `scale = clamp(sqrt(zoneArea / 1600), 0.6, 2.0)`, `zoneArea = mapSize² / число_зон × zone.size`; абсолюты × scale, perArea × sqrt(scale). Эталон 1600 ≈ типичная официальная зона.
- Пересчёт только в момент применения пресета (предсказуемо, не перетирает ручные правки) + кнопка «Пересчитать под текущую карту» у зоны/у всех зон.
- Динамический автопересчёт при изменении размера карты — НЕ делать (перетирает ручную работу втихую).

## 5. Что НЕ переносим от Aurora

- Глобальные проценты-слайдеры плотности (Structure/ResourceDensity) — у них это нужно, потому что генератор одноразовый; у нас значения зон редактируются прямо, проценты добавили бы вторую «скрытую» ручку поверх явных чисел.
- Эталон 6400 — заточен под их крупно-зонный генератор, берём свой (~1600).
